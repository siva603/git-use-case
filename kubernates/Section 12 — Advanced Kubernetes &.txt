# 12.1 Helm — Kubernetes Package Manager

Deploying large applications using raw YAML files can become difficult.

Example application may contain:

```

Deployment
Service
ConfigMap
Secret
Ingress
PersistentVolume

```id="helm-problem"

Managing all these files manually is complex.

Helm solves this problem.

Helm is the **package manager for Kubernetes**, similar to:

```

apt → Linux packages
npm → Node packages
helm → Kubernetes packages

```id="helm-comparison"

---

## Helm Architecture

```

Helm CLI
↓
Helm Chart
↓
Kubernetes Resources

```id="helm-architecture"

Helm packages applications into **Charts**.

Chart structure:

```

mychart/
├── Chart.yaml
├── values.yaml
├── templates/
└── charts/

````id="helm-chart-structure"

Explanation:

Chart.yaml  
chart metadata

values.yaml  
configuration values

templates  
Kubernetes YAML templates

---

## Installing Helm

Example command:

```bash id="install-helm-chart"
helm install myapp ./mychart
````

Check releases:

```bash id="helm-list"
helm list
```

Upgrade application:

```bash id="helm-upgrade"
helm upgrade myapp ./mychart
```

Rollback release:

```bash id="helm-rollback"
helm rollback myapp 1
```

Helm simplifies **application lifecycle management**.

# 12.2 Kustomize — Configuration Customization

Applications often need different configurations for environments.

Example:

````
dev environment
staging environment
production environment
``` id="env-config-example"

Instead of duplicating YAML files, Kubernetes provides **Kustomize**.

Kustomize allows:

• patching YAML  
• environment overlays  
• configuration customization

---

## Kustomize Architecture

````

Base Configuration
↓
Overlays (dev / prod)
↓
Final Kubernetes YAML

```id="kustomize-architecture"

Example structure:

```

k8s/
├── base/
└── overlays/
├── dev/
└── prod/

````id="kustomize-folder"

Generate manifests:

```bash id="kustomize-build"
kubectl apply -k overlays/prod
````

# 12.3 CI/CD with Kubernetes

In DevOps pipelines Kubernetes is used for **automated deployments**.

Typical CI/CD workflow:

````
Developer
    ↓
Git Repository
    ↓
CI Pipeline (Jenkins / GitHub Actions)
    ↓
Build Application
    ↓
Docker Image
    ↓
Push Image to Registry
    ↓
Helm Deployment
    ↓
Kubernetes Cluster
``` id="cicd-workflow"

This pipeline ensures **continuous delivery** of applications.

# 12.10 Troubleshooting Kubernetes

In production clusters issues can occur such as:

• application crashes  
• networking failures  
• node failures  
• control plane issues

Common debugging commands:

Check pods

```bash id="debug-get-pods"
kubectl get pods
````

Describe pod

```bash id="debug-describe"
kubectl describe pod <pod-name>
```

View logs

```bash id="debug-logs"
kubectl logs <pod-name>
```

Check events

```bash id="debug-events"
kubectl get events
```

Check node status

```bash id="debug-nodes"
kubectl get nodes
```
