Section 6 — Kubernetes Services & Networking

Pods in Kubernetes are **ephemeral**.

This means:

• Pods can crash  
• Pods can be recreated  
• Pod IP addresses can change

Example problem:

```

Pod A → 10.244.1.10
Pod B → 10.244.1.15

```

If Pod A crashes:

```

New Pod A → 10.244.1.21

```

Applications cannot rely on **changing IP addresses**.

Kubernetes solves this using **Services and Networking**.

This section explains:

• Kubernetes Service concept  
• ClusterIP  
• NodePort  
• LoadBalancer  
• Service Discovery  
• External DNS

---

# 6.1 What is a Kubernetes Service

A **Service** provides a **stable network endpoint** for a group of Pods.

Example architecture:

```

Service
│
┌─┼─────────┐
Pod1 Pod2 Pod3

```id="service-architecture"

The service automatically routes traffic to available Pods.

Clients communicate with the **Service**, not directly with Pods.

---

# 6.2 How Services Work

Services use **labels and selectors** to find Pods.

Example:

Pods:

```

Pod1 → label app=web
Pod2 → label app=web
Pod3 → label app=web

```id="pod-label-example"

Service selector:

```

selector:
app: web

```id="service-selector-example"

Traffic flow:

```

Client
↓
Service
↓
Pods (selected using labels)

```id="service-routing-flow"

---

# 6.3 Service Types

Kubernetes provides different service types depending on how applications should be exposed.

Main service types:

• ClusterIP  
• NodePort  
• LoadBalancer

---

# 6.4 ClusterIP Service

ClusterIP is the **default service type**.

It exposes the application **inside the Kubernetes cluster only**.

Architecture:

```

Internal Client
↓
ClusterIP Service
↓
Pods

````id="clusterip-architecture"

Example YAML:

```yaml id="clusterip-yaml"
apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 80
````

Create service:

```bash id="create-clusterip-service"
kubectl apply -f service.yaml
```

View services:

```bash id="list-services"
kubectl get services
```

Example output:

```
NAME          TYPE        CLUSTER-IP
web-service   ClusterIP   10.96.0.15
```

ClusterIP is used for **internal communication between microservices**.

---

# 6.5 NodePort Service

NodePort exposes the service on a **port of every cluster node**.

Architecture:

````
Client
   ↓
NodeIP:NodePort
   ↓
Service
   ↓
Pods
``` id="nodeport-architecture"

Example YAML:

```yaml id="nodeport-yaml"
apiVersion: v1
kind: Service
metadata:
  name: web-nodeport
spec:
  type: NodePort
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 80
    nodePort: 30007
````

Access application:

````
http://NodeIP:30007
``` id="nodeport-access"

Example:

````

[http://192.168.49.2:30007](http://192.168.49.2:30007)

```

NodePort range:

```

30000 – 32767

```

---

# 6.6 LoadBalancer Service

LoadBalancer services expose applications **externally using cloud load balancers**.

Architecture:

```

Internet
↓
Cloud Load Balancer
↓
Service
↓
Pods

````id="loadbalancer-architecture"

This is commonly used in cloud providers:

• AWS (ELB)  
• Azure Load Balancer  
• Google Cloud Load Balancer

Example YAML:

```yaml id="loadbalancer-yaml"
apiVersion: v1
kind: Service
metadata:
  name: web-lb
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 80
````

Get external IP:

```bash id="get-lb-ip"
kubectl get svc
```

Example output:

```
NAME      TYPE           EXTERNAL-IP
web-lb    LoadBalancer   34.120.5.12
```

Users access:

```
http://34.120.5.12
```

---

# 6.7 Service Discovery

Service discovery allows applications to **find each other automatically**.

Instead of using IP addresses, applications use **service names**.

Example:

````
http://web-service
``` id="service-dns-example"

Kubernetes DNS resolves the service name.

Example resolution:

````

web-service.default.svc.cluster.local

```id="dns-resolution-example"

Flow:

```

Application
↓
DNS Query
↓
Service IP
↓
Pod

````id="service-discovery-flow"

---

# 6.8 Kubernetes DNS (CoreDNS)

Kubernetes uses **CoreDNS** to provide service discovery.

CoreDNS runs as Pods inside the cluster.

Check DNS pods:

```bash id="check-coredns"
kubectl get pods -n kube-system
````

Example output:

```
coredns-558bd4d5db
```

CoreDNS automatically creates DNS records for:

• Services
• Pods

---

# 6.9 External DNS

External DNS automatically updates **DNS records for services**.

Example:

When a LoadBalancer service is created:

````
web-service → example.com
``` id="external-dns-example"

External DNS updates DNS providers such as:

• AWS Route53  
• Google DNS  
• Cloudflare

This allows external users to access services using domain names.

Example:

````

[https://myapp.example.com](https://myapp.example.com)

```

---

# 6.10 Networking Flow in Kubernetes

Complete networking flow:

```

Client
↓
Ingress / LoadBalancer
↓
Service
↓
Pods
↓
Containers

```id="full-network-flow"

Services act as **stable network entry points** for applications.
